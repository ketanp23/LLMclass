RAG Playground · Use case 02 · Structured data

Same question. Two columns. Two truths.

No documents here — just a warehouse with 248 tables and fourteen columns that could all plausibly mean "revenue." The model writes SQL, a validator inspects it, and it runs live against the data. Pick the wrong column and you still get a number: confident, precise, formatted to the rupee, and wrong.

How this works — read before you touch the knobs 6 steps

You are Priya, store manager, Mumbai 14. Your co-pilot sits on top of the retailer's warehouse — sales, targets, inventory, returns, footfall, refreshed hourly, governed by the data team. Nothing gets copied into the model. The pipeline learns the warehouse's shape, then writes a query against it. Six steps, and you control four.

STEP 01 · OFFLINE

Read the warehouse as it is

A schema reader walks the DDL: table names, column names, sample values, how tables join. It copies no rows. Queries will run live against the real thing, so answers are never stale.

The lesson248 tables, 4,800 columns, and 14 of them look like "revenue." That ambiguity is the entire problem in structured RAG.

STEP 02 · OFFLINE · YOURS

Build the semantic layer

Finance and data sit down once and agree: "revenue" means one column with one set of filters. "Slow-moving" means specific numbers. About 120 terms cover 95% of questions. The Semantic layer switch decides whether the model gets them.

Watch thisOff, the model resolves "revenue" by name similarity — the same way you would, skimming a column list. It picks something reasonable. It picks wrong.

STEP 03 · OFFLINE · YOURS

Make the schema retrievable

Table descriptions, the 120 definitions, the join graph, and ~800 vetted past question→SQL pairs are embedded into a vector store. You cannot paste 4,800 columns into a prompt — so the system retrieves only the handful the question needs.

Watch thisDrop Schema chunks retrieved too low and a table the question depends on never arrives. The model can't query what it can't see, and says so.

STEP 04 · LIVE · YOURS

Priya asks, on her phone

Plain English, the way she'd ask a colleague. Her identity comes attached: store_14, region west. A complex question ("why did I underperform?") is decomposed into sub-questions, because three small queries debug better than one giant one.

Watch thisIdentity is not a suggestion to the model — it becomes a WHERE store_id = 14 predicate that the validator refuses to run without.

STEP 05 · LIVE · YOURS

Generate SQL, then gate it

The model writes SQL using only the retrieved schema. Then the validator — plain code, no model — checks it: read-only, scope predicate present, tables on the allow-list, cost within cap. Only then does it reach the warehouse.

Watch thisClause colours show where each condition came from: definition, identity, the model's own guess.

STEP 06 · LIVE

Rows become a narrative

The result rows go back to the model with one instruction: explain these numbers to a store manager, cite the tables and definitions, show the SQL. It may rank, summarise and recommend — it may not invent a figure.

The lessonEvery number in the answer maps to a cell in the result table. If finance disputes it, the SQL is one tap away — and identical for every store manager who asks.

Four experiments — run them in order

  1. Kill the semantic layer. Ask the revenue question, then switch Semantic layer off. The model resolves "revenue" to a column that is genuinely named revenue_inr — and reports that Priya beat her target. The canonical definition says she missed it by 12.5%. Same question, opposite conclusion, no error message. This is the single most important step in structured RAG.
  2. Starve the retriever. Semantic layer back on. Drag Schema chunks retrieved down to 3. sales_fact — the table holding actual sales — falls out of the prompt, so the question becomes unanswerable. Compare this failure with the one above: this pipeline knows it failed. That one didn't.
  3. Remove the scope filter. Ask the Bangalore question with Identity scope on, then off. With the validator on, the query is rejected for having no scope predicate. Turn the validator off too and Priya reads another region's numbers — a governance breach in one click.
  4. Try to break the warehouse. Pick the last preset. The model happily writes a DELETE. The validator — not the model's good manners — is what stops it.

What's real here: the warehouse is ~2,400 generated rows in memory, and every figure on screen is computed from them at query time. The SQL you see is printed from the same query plan that executes, so the two cannot drift apart. Schema retrieval uses TF-IDF cosine similarity over the table and definition descriptions. The narration is a template that can only read cells from the result table — exactly the constraint a well-prompted LLM operates under.
